# 仕様と実装を形式手法で突き合わせるガイド

この book は、アプリケーションエンジニアが形式手法を実務に入れるための
導入ガイドとして構成する。

仕様や docs がある場合はそれを期待仕様として置き、コードとの矛盾を探す。
仕様がない、または信用できない場合は、既存実装・設定・テスト・運用上の
期待を de-facto 仕様として読み取る。

どちらの場合も、機械で反例を探し、最後にドメインの言葉へ戻して、
仕様を直すのか、コードを直すのか、両方を直すのかを確認する。

## AI と人間の境界

この book では、AI を「正しさを決める主体」として扱わない。
AI は、コード・docs・テストから候補となる claim を抜き、モデル案を作り、
機械検査の結果をドメイン語へ翻訳する作業者である。

人間が持つべき責任は、そもそも何を保証したいのか、
その性質が証明・検査できる形になっているのか、
反例が出たときに仕様と実装のどちらを直すのかを決めることである。

## 読み順

1. モデル化できる問いとして考える
2. 仕様・実装・ドメインを突き合わせる流れを掴む
3. 形式手法を採用するときの考え方
4. やりたいことからツールを選ぶ
5. ツールごとに何が証明・検査できるかを知る
6. 実例を見て、モデルからドメイン語へ戻す流れを掴む
7. モデルを作った後、仕様・コード・モデルの drift を維持する流れを掴む
8. 言語別チュートリアルとレシピで、実際の書き始め方を確認する

## この book の立場

形式手法は「全部を証明する」ために使わない。
実務では、次のような曖昧で壊れやすい作業を置き換えるために使う。

| 置き換えたい作業 | 形式手法で得るもの |
| --- | --- |
| レビューでの「たぶん大丈夫」 | 反例、または反例がないという機械検査結果 |
| 手書きの境界値テスト表 | 全入力に対する述語検査 |
| 設計会議のホワイトボード図 | 実行可能な関係モデル / 状態遷移モデル |
| 認可や config の暗黙仕様 | ドメイン語で読める契約 |
| refactor の勘 | 旧実装と新実装の等価性 / 差分 witness |

最終成果物は証明そのものではなく、次の台帳である。

```text
期待仕様 / 実装の主張 -> 機械検査 -> 反例または契約 -> ドメイン判断
```
