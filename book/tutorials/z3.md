# Z3 / SMT-LIB

Z3 は、実装や config から抜いた純粋述語に対して、
「この条件を満たす入力は存在するか」を聞く道具である。

時間順序や retry の全 interleaving ではなく、ある一瞬の入力組み合わせを見る。

## 最小チュートリアル

題材: checkout validator。

ドメイン語の claim:

```text
デジタル注文は total > 0 かつ email_len > 0 のときだけ valid。
未知の kind は reject。
```

SMT-LIB では、業務語を小さい型に落とす。

```smt2
(set-logic QF_LIA)

(define-fun physical_kind () Int 0)
(define-fun digital_kind () Int 1)

(define-fun valid_checkout
  ((kind Int) (has_shipping Bool) (email_len Int) (total Int))
  Bool
  (and
    (> total 0)
    (or
      (and (= kind physical_kind) has_shipping)
      (and (= kind digital_kind) (> email_len 0)))))
```

bad case は逆向きに聞く。

```smt2
(declare-const email_len Int)
(declare-const total Int)
(declare-const has_shipping Bool)

(assert (valid_checkout digital_kind has_shipping email_len total))
(assert (<= email_len 0))
(check-sat)
```

`unsat` なら「empty email で valid になる digital order は存在しない」。
`sat` なら witness を bug report にする。

repo の実行:

```sh
./languages/z3/check_checkout_form.sh
```

期待出力:

```text
unsat
unsat
sat
sat
sat
```

## 出力の読み方

| 出力 | 意味 |
| --- | --- |
| `sat` | 条件を満たす入力が存在する。`get-model` を付ければ具体値を取れる |
| `unsat` | 条件を満たす入力は存在しない |
| `unknown` | solver が答えられない。理論を単純化する、量化を減らす、encoding を変える |

`unsat` だけで安心しない。述語を間違えて全拒否にしても bad case は `unsat` になる。
必ず positive case と broken variant を入れる。

## レシピ

### config の dead rule 検出

置き換える作業:

- spreadsheet / YAML review で「この rule は誰かに当たるはず」と確認する作業

モデルにする入力:

- country
- plan
- appVersion
- allowlist / denylist
- env

検査する性質:

```text
enabled(config) && target(user, config)
```

期待する出力:

- `sat`: 対象 user の witness がある
- `unsat`: dead config。誰にも当たらない

### denylist が allowlist より強いことを lock

bad case:

```text
user in allowlist
user in denylist
isAllowed(user) == true
```

期待:

- `unsat`

`sat` なら priority 実装が壊れているか、仕様が曖昧。

### 旧 evaluator と新 evaluator の差分検出

モデル:

```text
old(user, config) != new(user, config)
```

期待:

- refactor が完全互換なら `unsat`
- 挙動変更を含むなら `sat` witness を release note / migration note に使う

### 認可 predicate の穴探し

bad case:

```text
allowed(user, ReadProject, project)
and user.org != project.org
```

期待:

- tenant isolation が正しければ `unsat`
- `sat` なら cross-tenant read の反例

## 避ける使い方

- queue / retry / crash の順序を Z3 だけで頑張る
- DB schema や framework の細部まで入れる
- `unsat` だけ見て positive case を置かない
- broken variant なしで CI green を信じる
