# 2. 形式手法を採用するときの考え方

形式手法は、アプリケーション全体を一気に証明するためのものではない。
まず「どの不安を、どの機械検査に置き換えるか」を決める。

前の章で見たように、目的は自然言語仕様を後から無理に翻訳することではない。
仕様を考える段階で、述語・関係・状態・遷移・不変条件・反例として見える
粒度に分解することから始める。

## 基本姿勢

実務では、仕様書がある場合とない場合で入り口が変わる。
仕様書や docs が信頼できるなら、それを期待仕様としてコードとの矛盾を探す。
仕様がない、または信用できないなら、実装を de-facto 仕様として読む。

どちらの場合も、入力は次のような実物になる。

| 入力 | 抜き出すもの |
| --- | --- |
| 仕様書 / docs / ADR | 期待されている業務ルール、禁止状態、保証 |
| 実装コード | branch guard、validator、状態遷移、fail-open / fail-close |
| テスト | 境界値、期待されている業務ルール |
| config | 条件の連言、優先順位、default 値 |
| 運用知識 | 「これは絶対に起きないはず」という暗黙前提 |
| incident | 再発してほしくない bad state |

## 3段の進め方

1. **吸い出す**: 期待仕様と実装が現に主張している仕様を分けて読む。
2. **反例探索する**: その主張を Z3 / Alloy / TLA+ / P / Dafny などに食わせる。
3. **突き合わせる**: 出た反例をドメインの言葉に戻して「どちらを直すか」と確認する。

反例が出たら、それは bug report 候補であり、同時に仕様確認の質問でもある。

## 採用すべき場所

形式手法が効くのは、テストが苦手なところである。

| 向く場所 | 理由 |
| --- | --- |
| config / policy / validator | 入力空間が広く、矛盾や抜けが多い |
| 認証・認可 | role、tenant、resource、override の組み合わせ爆発がある |
| 非同期 / retry / queue | 順序、重複、遅延、crash が bug を作る |
| state machine | 「この状態には行けないはず」を検査しやすい |
| refactor | 旧実装と新実装の差分 witness を取れる |
| data structure / domain invariant | 実装表現が変わっても守る抽象契約を置ける |

逆に、UI の見た目、曖昧な UX、統計的な良し悪し、外部サービスの品質評価は
最初の対象にしない。

## 成果物の形

目指すのは「証明の山」ではない。

```text
claim:
  デジタル注文は email が空なら valid にならない

machine check:
  Z3: digital empty email valid? -> unsat

domain wording:
  デジタル注文は email 必須。これは API 契約として lock されている。

next action:
  意図通りなら仕様に書く。意図でないなら実装を直す。
```

この形式にすると、形式手法の結果をドメイン owner、アプリケーション
エンジニア、レビュアが同じ土俵で扱える。
